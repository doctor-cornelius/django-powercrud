# Phase 2 Focused DaisyUI Override Points Notes

## Purpose

These notes support [`phase2-override-points-plan.md`](phase2-override-points-plan.md), which implements Phase 2 of the [`template_packs-master-plan.md`](../template_packs-master-plan.md).

Phase 2 turns the ratified focused-override contract into implementation. It does not introduce pack selection or JavaScript adapters; those are later master phases.

## Binding Contract

1. The Phase 1.2 contract defines the eleven supported override areas, their minimum context, semantic hooks, capabilities, and legacy bridges in [`phase1.2-1.5-template-pack-contract-analysis.md`](../phase1_contract/phase1.2-1.5-template-pack-contract-analysis.md).
2. Existing `powercrud/daisyUI/...` paths and direct server-addressable fragments remain available through 0.x.
3. A slice preserves default DaisyUI rendering and behaviour; focused overrides change presentation without copying PowerCRUD JavaScript.
4. Existing tests are the starting safety net. A slice adds a test only when it exposes an otherwise uncovered rendering, fragment, hook, or interaction contract.
5. `pcrud_mktemplate` retains legacy root and whole-tree copying through 0.x. Its focused-copy workflow is extended only for components already extracted and released.

## Delivery Rule

Every slice follows:

```text
characterize selected area -> extract one focused component set -> verify unchanged default behaviour -> document/copy the released component -> merge -> start the next area from the updated integration base
```

The Phase 2 child plan governs all focused override areas. Pagination is first because it is bounded and has no current JavaScript ownership; it is not a replacement for the remaining Phase 2 work.

## Plan

### Phase 2.0: Apply Shared Slice Safeguards

Use this phase as the standing acceptance rule for every implementation slice. Record only selected-area evidence here or in the relevant slice handover; do not turn it into a second broad template audit.

Phases 2.4–2.9 use a standing autonomous delivery protocol accepted after Phase 2.3 integration. Each numbered phase starts from synchronized `staging/main`, uses a numbered `template_pack/2.x` branch, and is divided into separately mapped, tested, documented, committed, and pushed sub-slices. Every sub-slice receives two fresh read-only maps: one for the exact runtime/template contract and one for existing coverage and genuine gaps. The main agent alone edits, tests, integrates, and judges compatibility.

At a numbered phase boundary, the full canonical suite must pass before the branch is soft-reset to its `staging/main` base and replaced with one semantic integration commit. That rewritten feature tip is not pushed. Local `staging/main` is fast-forwarded and pushed, then the completed feature branch is deleted locally and remotely before the next phase begins.

Routine work continues without user intervention. A material public-contract decision, non-trivial conflict, unexplained regression, compatibility risk, unrelated worktree change, or unexpected movement of remote `staging/main` is a stop condition. When the first genuine blocker occurs, create `phase2-blockers.md`, link it from the plan and these notes, and record its evidence, affected scope, safe options, independent work that may continue, and exact guidance required. Do not create an empty register pre-emptively.
