# Phase 9.3.1 Post-Cleanup Documentation Facts Audit

## Status

Complete as a read-only audit on `template_pack/9.3`. No product code, stable documentation, tests, templates, assets, or packaging files were changed. The findings below are the input to Phase 9.3.2.

## Purpose and evidence boundary

This audit confirms the permanent DaisyUI and Bootstrap behaviour that the five-page Template Packs section may document. Current implementation, tests, scripts, packaging configuration, and the accepted Phase 9 planning decisions are the evidence sources. Historical planning documents remain historical evidence and are not active product guidance.

The audit did not run containers, tests, builds, dependency installation, browsers, or sample servers. It used read-only source/document inspection and repository-wide searches. Every delegated finding was independently reviewed by the primary agent before inclusion.

## Executive findings

1. The permanent pack set is present and internally covered: DaisyUI is the implicit compatible default; Bootstrap is an explicit, process-startup selection with its own app, assets, templates, adapter, native forms, and `bootstrap5` Crispy integration.
2. The accepted five-page section is not yet present. Only `template_packs/index.md` exists; the four Phase 9.3.2 destination pages and their navigation entries remain to be written.
3. The current overview is a useful contract summary, but setup, customization precedence, third-party declaration mechanics, validator operation, and installed-artifact evidence are not yet documented in enough detail for an implementer or pack author.
4. Several active pages contain stale or overly DaisyUI-specific wording: the lowercase legacy `POWERCRUD_CSS_FRAMEWORK` example, the old styling-stack interpretation of that setting, the hard-coded Crispy Tailwind wording in form-field copy guidance, DaisyUI-only inline-edit wording, and the legacy `templates_path` default expression.
5. No active `daisyui-reference` or retired reference-presentation residue was found in `docs/mkdocs`, `src/powercrud`, `src/tests`, or `scripts`. Historical Phase 5/8 planning references remain valid and should be preserved.
6. The current third-party authoring boundary must be stated explicitly. Module-path declarations are supported, but the shipped validator and installed-artifact harness currently recognize only the DaisyUI and Bootstrap adapters/Crispy integrations, and the harness selectors are first-party-specific. The authoring guide must not promise arbitrary new framework adapters, variant adapters, Crispy integrations, or a generic third-party artifact gate without a separately approved tooling/API change.

## Phase 9.3.2 handoff

- `index.md`: retain the overview and make terminology, adapter ownership, startup-only selection, API classifications, Crispy ownership, and explicit-support rules authoritative.
- `selecting-and-configuring.md`: add DaisyUI/Bootstrap settings, app installation, base-template ownership, manual/Vite assets, native/Crispy setup, and sample launch links.
- `customising.md`: add resolution precedence, focused/model-scoped overrides, selected-pack copy behavior, legacy façade, whole-tree deprecation, and explicit custom ownership.
- `authoring-and-publishing.md`: document the shipped declaration fields, namespaces, capabilities, fragments, adapters, forms, assets, Hatch packaging, and the current existing-adapter/Crispy boundary. Do not promise arbitrary new framework or variant adapters.
- `testing-and-acceptance.md`: document validator checks, shared server/browser matrices, pack-specific sentinels, and isolated wheel/sdist resource validation. State that the current installed harness is first-party-selector-specific unless separately extended.

## Decision register

1. **Third-party scope:** The shipped code permits module-path declarations, but current adapter/Crispy validation and installed-artifact tooling are limited to the permanent first-party identities. Phase 9.3.2 should document that current supported authoring path explicitly and must not imply a public custom-adapter registry or generic artifact harness.
2. **Startup wording:** Say “configure before starting the process; no supported runtime/query/per-view switch.” Do not claim that selector resolution is permanently cached; tests prove settings/declarations are read dynamically.
3. **Whole-tree boundary:** A custom declaration may only expose whole-tree copying when it declares a valid one-segment legacy destination. Focused/model-scoped copying remains the normal route; a missing destination is the normal Bootstrap/third-party boundary.

## Phase 9.3.2 acceptance criteria

The next phase may begin when the five pages can be written without inventing facts or APIs, every F-01–F-22 row has an owning destination or an explicit “no action” disposition, stale active guidance is reconciled, and the current third-party scope is preserved. This audit itself is complete; stable documentation changes belong to Phase 9.3.2.
